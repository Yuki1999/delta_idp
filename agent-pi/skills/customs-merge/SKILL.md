# 多票合并申报（Customs Merge）Skill

## 使用场景

**当用户上传多张发票（`*_IV*.xlsx`）+ 多张箱单（`*_PL*.xlsx`）**，且明确要求合并成**一份**报关单时，必须使用本 Skill。

这是海关常见的"合并申报"场景：客户订单分批开票，但报关时归并为一份申报单。合并申报的合法前提是——所有发票的收发货人、贸易方式、运输方式一致；如果不一致，工具会通过 `check_consistency` 显式告警但仍允许强制合并。

---

## 必须遵循的工作流（不可跳步、不可自作主张）

### Step 1. 分别抽取每份发票和箱单

对用户列出的每份 `_IV` 文件调用一次 `read_invoice(file_path)`；对每份 `_PL` 文件调用一次 `read_packing_list(file_path)`。

**可以并行调用**：在一次响应中同时列出 N 个 `read_invoice` 和 M 个 `read_packing_list`，pi 会并行执行，速度更快。

每次调用会返回一段简短摘要（发票号、明细行数、数量/金额合计），完整 JSON 已经写入本次会话上下文，你不需要记住细节。

### Step 2. 校验一致性

抽完所有单据后，调用一次 `check_consistency()`（**无参数**）。

工具会自动读取上下文里已累积的所有发票和箱单，返回：
- 是否可以合并（`ok`）
- IV/PL 配对结果
- 主表关键字段（收发货人、贸易术语等）是否存在分歧
- 每对 IV/PL 的数量总计是否吻合

如果有告警：
- **主表字段分歧**：继续走 merge，工具会用 `/` 拼接分歧值，并在合并结果里保留告警
- **IV/PL 未配对** 或 **数量不吻合**：仍继续 merge，但要在最终输出里明显标注问题

### Step 3. 合并

调用一次 `merge_declarations()`（**无参数**）。

工具会：
- 主表字段：一致取任一 / 分歧用 `/` 拼接
- 数字类字段（数量、金额、毛净重、体积、箱数）：**精确累加**，由 JS 完成，不许你自己算
- 商品明细：所有票的明细纵向拼接，每行标注 `source_invoice`
- IV/PL 交叉引用：把 PL 里的 `ctn_no / net_kg / gross_kg` 按 `samsung_pn` 匹配拼到对应 IV 明细行上

### Step 4. 渲染

调用一次 `render_declaration()`（**无参数**）。

工具返回最终的两张 Markdown 表格（报关单主要信息 + 商品明细），加上一致性告警（如有）。**这就是给用户看的最终产物**——你只需要把这段 Markdown 原样呈现给用户即可，可以在前面写一两句总结（如"已合并 3 张发票"），但不要重新生成表格。

---

## 严格禁令

1. **绝对禁止自己做加法**。金额、毛净重、体积、件数的合计必须来自 `merge_declarations` 返回的数字，不许自己在回复里累加。LLM 加大数容易出错，工具用 JS 算保证精确。
2. **绝对禁止跳过工具直接输出报关单表格**。哪怕你觉得自己能一次做完，也必须走 `read_invoice → read_packing_list → check_consistency → merge_declarations → render_declaration` 全流程，用户需要看到每一步的 tool 轨迹作为审计凭据。
3. **绝对禁止只处理部分文件**。用户提供了 N 份 IV + M 份 PL，就必须对每一份都调用一次 read。漏掉任何一份都会导致合并结果错误。
4. **绝对禁止编造字段值**。工具返回什么就是什么；如果某个字段是 null 或缺失，render_declaration 会显示空白，这没问题，不要用推测的值填充。

---

## 合并规则速查（供你理解工具行为）

| 类型 | 规则 |
|---|---|
| 发票号、日期 | 用 `/` 拼接：`INV001 / INV002 / INV003` |
| 收发货人、贸易术语、目的港、原产地、币制、商品名 | 三票一致时取任一；分歧时 `/` 拼接并在告警里显示 |
| 总数量、总金额、总毛重、总净重、总体积、总箱数 | **精确累加** |
| 商品明细 | 纵向拼接所有行；每行加 `source_invoice` 列；`ctn_no / net_kg / gross_kg` 从对应 PL 拉入 |

---

## 何时不用本 Skill

- 用户只上传了 **一张** IV + 一张 PL（单票业务）→ 用现有的 `doc-extraction` skill 走单票流程即可
- 用户明确说 "分别抽三份、不要合并" → 逐个走单票流程
- 上传文件里根本没有 `_IV` / `_PL` 后缀识别不出 → 询问用户先明确文件类型
